# システム仕様

本アプリは開発環境のパッケージマネージャーおよびエディタ拡張・設定ファイルをバックアップ／リストアする Electron デスクトップアプリである。

## ストレージ構成

### アプリ設定ディレクトリ (`~/.abr`)

バックアップデータとは別に、アプリ自身の設定をユーザーホーム配下に保持する。

- ルート: `~/.abr` (`APP_DIR_NAME = '.abr'`)。アプリ起動時に `ensureAppDirectories()` で自動生成する。
- 設定ファイル: `~/.abr/config.json`。スキーマは以下。

| キー | 型 | 説明 |
| --- | --- | --- |
| `backupDirectory` | `string` | ユーザーが選択したバックアップ先の絶対パス。未選択時は空文字 |
| `language` | `'ja' \| 'en'` | UI 言語 |
| `darkMode` | `boolean` | ダークモードの有無 |

- 初回起動 (`language` 未設定または `darkMode` が `undefined`) のときのみ、OS のロケール (`app.getLocale()`) とテーマ (`nativeTheme.shouldUseDarkColors`) から `language` / `darkMode` を自動判定して `config.json` に書き戻す。ロケールが `ja` で始まる場合のみ `ja`、それ以外は `en` にフォールバックする。

### バックアップディレクトリ (ユーザー選択)

実際のバックアップ成果物は、ユーザーが選んだ `backupDirectory` 配下に格納する。アプリ設定ディレクトリとは独立している。

```text
${backupDirectory}/
├── ${managerId}_apps.json     # パッケージマネージャーごとのバックアップ (JSON 配列)
├── ${vscodeId}/               # VS Code 系エディタごと (1 ディレクトリ)
│   ├── extensions.json        #   拡張一覧
│   ├── extensions_wsl.json    #   WSL 内拡張一覧 (Windows かつ検出時のみ)
│   ├── settings.json          #   設定ファイル群 (basename のみで平坦化)
│   ├── keybindings.json
│   └── ...                    #   mcp.json / tasks.json など定義依存
├── ${configAppId}/            # 設定ファイル系アプリごと (1 ディレクトリ)
│   └── ...                    #   元ファイルを basename で配置
└── .cache/
    └── winget_cache.json      # winget 表示名キャッシュ
```

- バックアップの「最終更新日時」は、単一ファイル形式 (パッケージマネージャー) はその JSON の mtime、ディレクトリ形式 (VS Code 系・設定アプリ) は配下を再帰探索した最新 mtime を ISO 8601 文字列で返す。存在しなければ `null`。
- パッケージマネージャーのバックアップはメタデータを持たない素の JSON 配列であり、別途の `backup_metadata.json` は使用しない。

## バックアップ・リストア対象

### パッケージマネージャー

| OS | 対象 |
| --- | --- |
| Windows (`win32`) | winget, msstore (Microsoft Store), scoop, chocolatey |
| macOS (`darwin`) | homebrew |
| Linux | homebrew, apt, yum, dnf, pacman, zypper, snap, flatpak |

- 定義は `MANAGER_DEFS` (`src/shared/constants.ts`)。各定義は `id` / `label` (UI 表示) / `os` (対応プラットフォーム) を持つ。
- 一括バックアップ時の OS 既定対象は、Windows = winget/msstore/scoop/chocolatey、macOS = homebrew、Linux = apt/yum/dnf/pacman/zypper/snap/flatpak。
- マネージャーの利用可否判定 (検出) は `Promise.all` で全マネージャーを並列に確認する。

### VS Code 系エディタ

`VS_CODE_DEFS` で定義。対象 ID: `vscode` / `cursor` / `antigravity-ide` / `voideditor`。

- バックアップ対象は (1) 拡張一覧、(2) 標準設定ファイル (`User/settings.json`, `User/keybindings.json`)、(3) 定義依存のアプリ固有ファイル (例: `mcp.json`, `tasks.json`, `mcp_config.json`, `GEMINI.md` など) で、いずれもバックアップ先には **basename のみで平坦化**して保存する (ディレクトリ階層は再現しない)。
- 検出方法はプラットフォーム依存:
  - macOS: アプリバンドル内の実行ファイル (`/Applications/<アプリ名>/Contents/Resources/app/bin/<command>`) の存在で判定する (PATH 検索ではない)。
  - Windows / Linux: コマンド名 (`code` / `cursor` / `void` 等) を PATH から検索する。
- 設定ファイルのルートパスはバックアップ／リストア実行時に展開する: macOS=`~/Library/Application Support/<App>`、Windows=`%APPDATA%\<App>`、Linux=`~/.config/<App>`。

### 設定ファイル系アプリ

`CONFIG_APP_DEFS` で定義。対象 ID: `bash_rc` / `zsh_rc` / `git_rc` / `npm_rc` / `pypi_rc` / `wsl_rc` / `powershell_rc` / `claude_desktop` / `claude_code` / `gemini_cli` / `tabby` / `filezilla` / `dbgate`。

- 各アプリはプラットフォームごとのファイルパス配列を持つ。パスは `resolveEnvPath()` で環境変数 (`%APPDATA%`, `%USERPROFILE%`, `~` 等) を展開してから解決する。
- 末尾が区切り文字 (`/` または `\`) のパスはディレクトリとみなし再帰コピー、それ以外は単一ファイルとしてコピーする。

## バックアップエンジン

- パッケージマネージャー: 各マネージャーの `list*()` を実行し、結果を `${managerId}_apps.json` に JSON 配列で書き出す。`identifiers` を渡すと特定パッケージのみに絞り込む。
- 出力スキーマはマネージャーごとに異なる (`src/shared/types.ts`)。主な例:
  - winget / msstore: `{ PackageId, Name, Version }`
  - scoop: `{ Name, Version, Source? }`
  - chocolatey: `{ PackageId, Title, Version }`
  - homebrew: `{ Name, Version, InstalledOnRequest? }`
  - apt: `{ Package, Version, Architecture? }`
  - yum / dnf / zypper: `{ Name, Version, Release?, Architecture? }`
  - pacman: `{ Name, Version, Repository? }`
  - snap: `{ Name, Version, Revision?, Tracking? }`
  - flatpak: `{ Name, Application, Version, Branch?, Origin? }`

### winget 表示名キャッシュ

winget はパッケージ ID しか即時に得られないため、表示名 (Name) を補完する。

- キャッシュは `${backupDirectory}/.cache/winget_cache.json` に保存する (グローバルではなくバックアップ先ごと)。構造は `Record<packageId, { package_id, cached_at, display_name }>`。
- キャッシュミス時は `winget show` → `winget search` → `winget list` の順で表示名取得を試み、得られた結果を ISO タイムスタンプ付きでキャッシュする。いずれも失敗した場合はパッケージ ID をそのまま表示名として使う。
- 取得処理は 1 件ずつ逐次実行し、その進捗を `task:progress` (`key: 'gettingDisplayNames'`, `params: { source, current, total }`) で renderer に送る。進捗通知が出るのはこの winget 表示名取得のみである。

## リストアエンジン

- リストアは常に**逐次実行**であり、並列インストールは行わない (項目の順序が結果に影響する)。各インストールコマンドの完了を待ってから次へ進む。
- パッケージマネージャー: `buildInstallCommand()` でマネージャー別コマンドを生成して実行する (例: `winget install <id>`、`brew install <name>`、`sudo apt install -y <name>`)。
- VS Code 拡張: ローカルは `code --install-extension <id>` 等を直接実行。WSL は `runCommandInWSL()` 経由で `wsl -- <command> --install-extension <id>` を実行する。
- 設定ファイル: バックアップ先ツリーとリストア先を突き合わせて競合を検出し、`RestoreConflictItem` (相対表示パス、両者のサイズ・mtime) の配列を UI に返す。UI が選んだ `skipPaths` (相対パス) を除外してコピーする。リストア先ディレクトリが無ければ作成する。
- スクリプト出力: ファイルを作らず、1 行 1 インストールコマンドのシェルスクリプト文字列を生成して返す。パスに空白を含む場合は引用符で囲む。

## プラットフォーム別のコマンド実行

- macOS: 起動時に `fixPathOnMacOS()` が `zsh -l -c env` を実行してログインシェル環境 (PATH 等) を読み込み `process.env` に反映する。各コマンドは `/bin/zsh -c` 経由で実行し、シェル関数・エイリアスを継承する。
- Windows: 各コマンドは PowerShell (`spawn('powershell', ['-Command', ...])`) 経由で実行する。
- Linux: コマンドを直接 spawn する (PATH 上か絶対パスである必要がある)。

### WSL 連携 (Windows のみ)

- 検出: PowerShell で `Get-Command wsl` を実行して WSL の利用可否を判定する。
- 拡張一覧: 通常の拡張一覧とは別に `wsl -- <command> --list-extensions --show-versions` を実行し、検出された場合のみ `${vscodeId}/extensions_wsl.json` に保存する (通常拡張とは別ファイル)。
- リストア: `wsl -- <command> --install-extension <id>` を逐次実行する。

## IPC アーキテクチャ

- renderer ↔ main は `ipcRenderer.invoke()` / `ipcMain.handle()` を基本とし、進捗通知 (`task:progress`) と更新状態通知 (`updater:stateChanged`) のみ main → renderer の push (event) で送る。
- チャンネル定数は `IPC_CHANNELS` (`src/shared/constants.ts`) に集約。接頭辞ごとの分類: `config:*` (設定) / `detect:*`・`check:*` (検出) / `list:*` (一覧取得) / `backup:*` (バックアップ) / `restore:*` (リストア) / `app:*` (アプリ情報・設定) / `window:*` (ウィンドウ操作) / `updater:*` (自動更新)。
- preload (`src/preload/index.ts`) は `window.abr` グローバルを公開し、各メソッドを IPC チャンネルに 1:1 で対応付ける。すべて非同期で、進捗は `onTaskProgress()` のリスナー方式で受け取る。
- main 側のハンドラ登録 (`registerIpcHandlers()`) は `BackupManager` / `RestoreManager` のシングルトンに委譲し、両者がバックアップディレクトリの解決と低レベルサービスのラップを担う。
